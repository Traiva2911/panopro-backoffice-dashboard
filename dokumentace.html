AI Backoffice Agent — Dokumentace

AI agent, který automatizuje firemní backoffice PANOPRO: třídění e-mailů, CRM úkoly, účetní podklady a hlídání pohledávek. „Mozkem" je Claude, „rukama" Zapier.

Dashboard (produkční URL): https://black-mushroom-06036841e.7.azurestaticapps.net Repozitář dashboardu: https://github.com/Traiva2911/panopro-backoffice-dashboard (větev main) Kód agenta: lokální projekt C:\Users\amati\zapier-sdk (TypeScript, Node 24) Hosting dashboardu: Azure Static Web Apps (Free tier) + GitHub Pages (záloha)

Dokumentace má dvě části: Část A — Technická (správa, konfigurace, nasazení) a Část B — Pro Víťu (předvádění a prodej klientům).

ČÁST A — Technická dokumentace

1. Architektura

┌─────────────────────────────────────────────────────┐
│  VSTUP – schránky (Zapier SDK, "pull")              │
│  zapier.ts                                          │
│  • Gmail  (list_messages_v0)                        │
│  • Outlook (new_email_personal)                     │
│  → sjednocení do NormalizedEmail                    │
├─────────────────────────────────────────────────────┤
│  MOZEK – klasifikace (Claude API)                   │
│  classify.ts                                        │
│  Model: claude-opus-4-8 (structured outputs / Zod)  │
│  → kategorie, urgence, owner, data faktury, odpověď │
├─────────────────────────────────────────────────────┤
│  AKCE – podle kategorie (vše pod dryRun)            │
│  triage.ts + actions.ts + invoices.ts + receivables │
│  • koncepty odpovědí (Gmail/Outlook)                │
│  • úkoly + dealy (HubSpot REST API)                 │
│  • registr faktur (.state/invoices.json)            │
│  • upomínky 3 úrovní + eskalace                     │
├─────────────────────────────────────────────────────┤
│  VÝSTUP – dashboard                                 │
│  dashboard.ts → dashboard.html → Azure / GitHub     │
└─────────────────────────────────────────────────────┘

Datový tok (jeden běh / "tick")

  1. triage.ts načte přes zapier.ts nové e-maily z Gmailu i Outlooku (paralelně)
  2. state.ts odfiltruje ty, které agent už viděl (kurzor v .state/processed.json)
  3. Každý nový e-mail jde do classify.ts → Claude vrátí zvalidovaný objekt (Zod)
  4. triage.ts určí odpovědnou osobu (taxonomy.ts) a podle kategorie spustí akce
  5. Akce (actions.ts, invoices.ts, receivables.ts) respektují config.dryRun
  6. Stav se uloží; dashboard.ts vykreslí výsledky do dashboard.html

3. Moduly (4 oblasti zadání)

# Modul Soubory Co dělá
1 E-mailová agenda classify.ts, triage.ts Třídí (poptávka/faktura/urgence/reklamace/jiné), navrhuje odpovědi, přiřazuje osobě
2 CRM a obchod actions.ts, state.ts Dealy + úkoly v HubSpotu, watchlist nezodpovězených poptávek
3 Účetní agenda invoices.ts, excel.ts Vytahuje dodavatele/částku/splatnost/VS, hlídá chybějící doklad, páruje dle VS
4 Pohledávky receivables.ts, dunning.ts Upomínky 3 úrovní, eskalace neplatičů, report (aging, sumy)

Jak funguje klasifikace

Claude dostane od/předmět/přílohu/náhled textu a přes Zod schéma je donucen vrátit přesně tvarovaný objekt: category, urgency, summary, owner, invoice (jen u faktur), suggestedReply. Žádné parsování volného textu — co neprojde validací, spadne do bezpečného fallbacku (jine).

4. Pravidla a směrování (src/taxonomy.ts)

Tohle je „mozek pravidel" — doménová znalost, kterou Claude sám neví.

Konstanta Co určuje
PEOPLE Role → jméno + e-mail (zatím vše info@panopro.cz)
ROUTING Výchozí osoba podle kategorie
BUSINESS_CONTEXT Popis firmy PANOPRO (služby, segmenty) pro přesnější klasifikaci
URGENCY_HINTS Co znamená „vysoká urgence"

Přidání / změna odpovědné osoby

  1. Otevři src/taxonomy.ts
  2. Do PEOPLE přidej/uprav řádek klic: { name: "Jméno", email: "adresa@panopro.cz" }
  3. Případně uprav ROUTING, aby kategorie mířila na nový klíč
  4. Směrování se přizpůsobí samo — nic dalšího není potřeba

Úprava kontextu firmy

Uprav text v BUSINESS_CONTEXT (čím konkrétnější, tím přesnější třídění) nebo URGENCY_HINTS.

6. API klíč (.env)

8. Dashboard

Jak funguje

dashboard.ts vezme výsledky běhu a vykreslí je do samostatného dashboard.html (data „zapečená" dovnitř → funguje offline). Generuje se automaticky na konci npm run demo.

Sekce: KPI karty → e-mailová triage → registr faktur → pohledávky s agingem. Štítek vpravo nahoře (TESTOVACÍ REŽIM / OSTRÝ PROVOZ) se řídí podle config.dryRun.

Úprava vzhledu

Renderer má tři funkce v src/dashboard.ts: renderStyles() (CSS), renderDashboardBody() (obsah), renderDashboardHtml() (celý soubor). Logo = oficiální obrázek PANOPRO z panopro.cz (bílá verze s claimem) — stejný jako PANOPRO Advisor.

12. Bezpečnost

  • dryRun: true dokud není výstup ověřen — pojistka proti nechtěnému odeslání.
  • ANTHROPIC_API_KEY jen v .env (necommitovat); deployment token nikdy v kódu.
  • Dashboard je veřejný → momentálně jen demo data. Před nasazením ostrých dat přidat přihlášení (Azure SWA: Entra ID / heslo).
  • HubSpot zápis vyžaduje write scope na připojení.

ČÁST B — Pro Víťu (předvádění a prodej)

Lidský průvodce: co to je, co to řeší a jak to ukázat klientovi. Techniku řeší Andrea — ty potřebuješ rozumět hodnotě a umět to předvést.

B1. Co to je (jednou větou)

Chytrý asistent, který za firmu obhospodařuje příchozí e-maily, účetní podklady, CRM a pohledávky — sám je třídí, připravuje odpovědi a hlídá, co je potřeba. Nenahrazuje lidi — bere jim z krku nudnou ruční práci a říká, co vyžaduje pozornost.

B2. Co to řeší (tohle prodáváš)

Oblast Bolest klienta Co agent udělá
📨 E-maily „Chodí nám stovky mailů, něco zapadne." Roztřídí, připraví odpověď, pošle správné osobě
💼 CRM / obchod „Poptávky se ztrácejí." Založí obchodní případ a úkol, hlídá nezodpovězené
🧾 Účetnictví „Faktury přepisujeme ručně." Vytáhne dodavatele, částku, splatnost, VS; hlídá chybějící doklad
💰 Pohledávky „Zapomínáme upomínat." Hlídá po splatnosti, sám pošle upomínky a eskaluje

Prodejní věta: „Ušetří backoffice hodiny denně a nic důležitého nezapadne."

B3. Jak to ukázat klientovi

A) Dashboard (nejrychlejší): otevři odkaz https://black-mushroom-06036841e.7.azurestaticapps.net — přehledná palubní deska, funguje na mobilu i na plátně.

B) „Mozek" naživo (působivější, u Andrey): příkaz npm run demo ukáže, jak agent naživo třídí ukázkové e-maily. Bezpečné — nic se neodešle.

V dashboardu jsou teď ukázková data (smyšlené Alfa/Beta/Gama) — na demo ideální, žádná citlivá data ven nejdou.

B4. Mluvící body k dashboardu

Co na obrazovce Co řekneš
Karty nahoře (KPI) „Hned vidíš, kolik toho agent zpracoval a co hoří."
Poptávka → úkol + odpověď „Poptávku sám pozná, založí v CRM a připraví odpověď."
Faktura → částka, VS, splatnost „Z mailu vytáhne fakturační údaje pro účetní."
Červené „Chybí doklad" „Sám hlídá, když k faktuře chybí doklad."
Pohledávky → upomínky + eskalace „Hlídá neplatiče a sám stupňuje upomínky."

B5. Časté otázky klientů

  • „Nahradí nám to lidi?" → Ne. Bere jim ruční nudnou práci, lidi dělají rozhodnutí.
  • „Co naše data?" → Běží na firemních schránkách klienta, nic se neposílá navíc. Lze chránit přihlášením.
  • „Posílá to maily samo?" → Defaultně ne — připravuje koncepty, člověk je odentruje. Naostro až klient řekne.
  • „Jak dlouho trvá nasadit?" → Základ rychle; doladění pár hodin technické práce (Andrea).
  • „Funguje to česky?" → Ano, celé v češtině.

B6. Dělba práce

  • Ty (Víťa): sháníš klienty, předvádíš (odkaz + mluvící body), zjistíš potřeby.
  • Andrea: technické nasazení, napojení schránek/CRM, doladění, údržba.

Když klient řekne „ano", předáš Andree: jakou má schránku (Gmail/Outlook), jaké CRM, kdo co ve firmě řeší.
