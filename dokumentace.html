AI Backoffice Agent — Dokumentace

AI agent, který automatizuje firemní backoffice PANOPRO: třídění e-mailů, CRM úkoly, účetní podklady a hlídání pohledávek. „Mozkem" je Claude, „rukama" Zapier.

Dashboard (produkční URL): https://black-mushroom-06036841e.7.azurestaticapps.net Repozitář dashboardu: https://github.com/Traiva2911/panopro-backoffice-dashboard (větev main) Kód agenta: lokální projekt C:\Users\amati\ai-backoffice-agent (TypeScript, Node 24) Hosting dashboardu: Azure Static Web Apps (Free tier) + GitHub Pages (záloha)

1. Pokrytí zadání

Všechny čtyři oblasti zadání jsou implementované a otestované. U každé schopnosti je uvedeno, na jaké integraci je funkčnost ověřená a čím ji lze nahradit — agent není uzamčen na konkrétní nástroj.

Schopnost ze zadání Stav Otestováno na Lze napojit i
Třídění příchozích e-mailů ✅ Gmail + Outlook jiná schránka / IMAP přes Zapier

Přiřazení odpovědné osobě ✅ pravidla v taxonomy.ts libovolné role/adresy

4. Pohledávky

Schopnost ze zadání Stav Otestováno na Lze napojit i
Hlídání faktur po splatnosti ✅ registr faktur —
Automatické upomínky ✅ Gmail/Outlook koncepty (3 úrovně) jiná schránka
Eskalace neplatičů ✅ úkol v HubSpotu jiný CRM
Reporting pohledávek ✅ dashboard (aging, sumy, eskalace) export do Excelu / Sheets

Závěr porovnání se zadáním: nic nechybí. Všechny dílčí body jsou pokryté. Dvě věci jsou na roadmapě jako vylepšení (ne chybějící funkce): čtení faktur přímo z PDF příloh a volitelný denní e-mailový souhrn.

2. Integrace a výměna řešení

Agent je postavený modulárně: logika (mozek) je oddělená od konkrétních nástrojů (ruce). Nástroje se připojují přes Zapier konektory a REST API, takže je lze vyměnit bez přepisu logiky.

Vrstva Otestováno na Alternativy (lze napojit)
Schránky Gmail, Outlook další Google/Microsoft účty, IMAP schránky
CRM HubSpot Pipedrive, Raynet, Salesforce, jiný CRM s API/Zapier
Účetní podklady Excel (+ lokální registr) Google Sheets, Pohoda, Money, Fakturoid (přes API/export)
AI klasifikace Claude (Anthropic) jádro řešení

Důležité: funkčnost je ověřená na výše uvedených nástrojích (Gmail, Outlook, HubSpot, Excel, Claude). Napojení jiného řešení je otázka konfigurace konektoru, ne přepisu agenta — proto se dá nasadit i u klienta, který používá jiné systémy.

3. Architektura

┌─────────────────────────────────────────────────────┐
│  VSTUP – schránky (Zapier SDK, "pull")              │
│  zapier.ts                                          │
│  • Gmail  (list_messages_v0)                        │
│  • Outlook (new_email_personal)                     │
│  → sjednocení do NormalizedEmail                    │
├─────────────────────────────────────────────────────┤
│  MOZEK – klasifikace (Claude API)                   │
│  classify.ts                                        │
│  Model: claude-opus-4-8 (structured outputs / Zod)  │
│  → kategorie, urgence, owner, data faktury, odpověď │
├─────────────────────────────────────────────────────┤
│  AKCE – podle kategorie (vše pod dryRun)            │
│  triage.ts + actions.ts + invoices.ts + receivables │
│  • koncepty odpovědí (Gmail/Outlook)                │
│  • úkoly + dealy (HubSpot REST API)                 │
│  • registr faktur (.state/invoices.json)            │
│  • upomínky 3 úrovní + eskalace                     │
├─────────────────────────────────────────────────────┤
│  VÝSTUP – dashboard                                 │
│  dashboard.ts → dashboard.html → Azure / GitHub     │
└─────────────────────────────────────────────────────┘

Datový tok (jeden běh / "tick")

  1. triage.ts načte přes zapier.ts nové e-maily z Gmailu i Outlooku (paralelně)
  2. state.ts odfiltruje ty, které agent už viděl (kurzor v .state/processed.json)
  3. Každý nový e-mail jde do classify.ts → Claude vrátí zvalidovaný objekt (Zod)
  4. triage.ts určí odpovědnou osobu (taxonomy.ts) a podle kategorie spustí akce
  5. Akce (actions.ts, invoices.ts, receivables.ts) respektují config.dryRun
  6. Stav se uloží; dashboard.ts vykreslí výsledky do dashboard.html

5. Moduly

# Modul Soubory Co dělá Otestováno / lze vyměnit
1 E-mailová agenda classify.ts, triage.ts Třídí, navrhuje odpovědi, přiřazuje osobě Gmail/Outlook → jiná schránka
2 CRM a obchod actions.ts, state.ts Dealy + úkoly, watchlist poptávek, upozornění HubSpot → jiný CRM
3 Účetní agenda invoices.ts, excel.ts Faktury, chybějící doklad, párování VS Excel → Sheets/účetní systém
4 Pohledávky receivables.ts, dunning.ts Upomínky 3 úrovní, eskalace, report HubSpot/schránka → alternativy

Jak funguje klasifikace

Claude dostane od/předmět/přílohu/náhled textu a přes Zod schéma je donucen vrátit přesně tvarovaný objekt: category, urgency, summary, owner, invoice (jen u faktur), suggestedReply. Žádné parsování volného textu — co neprojde validací, spadne do bezpečného fallbacku (jine).

6. Pravidla a směrování (src/taxonomy.ts)

Konstanta Co určuje
PEOPLE Role → jméno + e-mail (zatím vše info@panopro.cz)
ROUTING Výchozí osoba podle kategorie
BUSINESS_CONTEXT Popis firmy pro přesnější klasifikaci
URGENCY_HINTS Co znamená „vysoká urgence"

Přidání / změna odpovědné osoby

  1. Otevři src/taxonomy.ts
  2. Do PEOPLE přidej/uprav klic: { name: "Jméno", email: "adresa@panopro.cz" }
  3. Případně uprav ROUTING; směrování se přizpůsobí samo

10. Dashboard

dashboard.ts vykreslí výsledky běhu do samostatného dashboard.html (data „zapečená" dovnitř → funguje offline). Generuje se na konci npm run demo. Sekce: KPI karty → triage → registr faktur → pohledávky s agingem. Štítek TESTOVACÍ REŽIM / OSTRÝ PROVOZ se řídí podle config.dryRun. Logo = oficiální obrázek PANOPRO z panopro.cz (stejný jako PANOPRO Advisor).

14. Bezpečnost

  • dryRun: true dokud není výstup ověřen — pojistka proti nechtěnému odeslání.
  • ANTHROPIC_API_KEY jen v .env; deployment token nikdy v kódu.
  • Dashboard je veřejný → momentálně jen demo data. Před ostrými daty přidat přihlášení (Azure SWA: Entra ID / heslo).
  • Zápis do CRM vyžaduje write scope na připojení.

15. Roadmap

  1. Ověřit, že PANOPRO maily chodí do čtené schránky (jinak vyměnit connectionId).
  2. Denní souhrn do e-mailu (volitelné rozšíření bodu „upozornění na důležité e-maily").
  3. Čtení PDF příloh u faktur (rozšíření „vytahování faktur").
  4. Aktivovat Excel/Sheets zrcadlo pro účetní.
  5. Přepnout dryRun: false + naplánovat běh (~15 min, Plánovač úloh Windows).
  6. Dashboard: přidat přihlášení, auto-deploy z GitHubu, volitelně doména dashboard.panopro.cz.
